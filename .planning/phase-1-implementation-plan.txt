TASK 2 — Stripe Checkout Route Handler
File: app/api/checkout/route.ts

What: POST endpoint that creates a Stripe Checkout Session and returns the
redirect URL.

Logic:
  1. Parse quantity (seats) from request body; validate it is >= 1
  2. Create Checkout Session via Stripe SDK:
       mode: 'subscription'
       line_items: [{ price: 'price_1ThbLNCzT2268ei9nkadS8kD', quantity }]
       adjustable_quantity: { enabled: true, minimum: 1 }
       success_url: '/onboarding?session_id={CHECKOUT_SESSION_ID}'
       cancel_url: '/'
       tax_id_collection: { enabled: true }
  3. Return { url: session.url }

Env needed: STRIPE_SECRET_KEY (already in .env.local)

TASK 3 — Stripe Webhook Handler
File: app/api/webhooks/stripe/route.ts

What: Handles Stripe's checkout.session.completed event — this is where a
paid firm gets created in the database.

Logic:
  1. await req.text() for raw body; verify signature with
     stripe.webhooks.constructEventAsync()
  2. Only handle checkout.session.completed; return 200 for all others
  3. Check processed_stripe_events — if event already there, return 200
     (idempotent)
  4. In a single Postgres transaction (via service-role client):
       - Insert into firms: { name: 'Pending', stripe_customer_id,
         stripe_subscription_id, seats: quantity, status: 'active' }
       - Insert into firm_members: { firm_id, user_id: null,
         email: admin_email, role: 'firm_admin', status: 'invited' }
       - Insert into processed_stripe_events: { event_id }
  5. Fire a fire-and-forget POST to /api/send-welcome-email (Task 4b)
     passing { email, firm_id } — do not await it inline

Env needed: STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, SUPABASE_SERVICE_ROLE_KEY

Requirements covered: PAY-01, PAY-02, PAY-03, PAY-04, PAY-06, PAY-07

TASK 4B — Welcome / Magic-Link Email Sender
File: app/api/send-welcome-email/route.ts

What: Called fire-and-forget from the webhook handler. Sends the magic-link
invite to the firm admin.

Logic:
  1. Use Supabase Auth admin API:
     auth.admin.generateLink({ type: 'magiclink', email })
     → get the magic link URL
  2. Call Resend REST API (POST https://api.resend.com/emails) with:
       From: noreply@aistaffcompliance.com
       To: admin email
       Subject: "Set up your AI Compliance Training account"
       Body: plain welcome message + magic link button
  3. Return 200

Env needed: RESEND_API_KEY, SUPABASE_SERVICE_ROLE_KEY

TASK 5 — Auth Callback Handler
File: app/auth/confirm/route.ts

What: Handles Supabase's magic-link redirect (the ?token_hash=&type= URL
that Supabase emails).

Logic:
  1. Parse token_hash and type from query params
  2. Call supabase.auth.verifyOtp({ token_hash, type }) to exchange the token
     for a session
  3. Set firm_id and role into app_metadata via service-role
     auth.admin.updateUserById() — reads from firm_members table by email
  4. If type is 'invite', redirect to /accept-invite
     Otherwise redirect to /dashboard on success, / on error

Note: This is the standard Supabase SSR auth callback pattern — @supabase/ssr
docs show this exactly.

Requirements covered: AUTH-01, AUTH-02

TASK 10 — Cert Worker: Full PDF Generation
File: workers/cert-worker/src/index.ts

What: Fill in the 3 TODO stubs in the existing cert worker. Triggered by
Supabase DB Webhook on quiz_attempts INSERT where passed = true.

Steps inside the Worker:
  1. Fetch employee name + firm name from Supabase (service-role)
  2. Generate cert ID: BSBR-{year}-{XXXX}-{XXXX}
     (sequential — uses DB sequence already defined in 0001_initial_schema.sql)
  3. Build PDF with pdf-lib:
       - Recipient name
       - Course name + version
       - Issue date
       - Expiry date (issue date + 365 days)
       - Score
       - Firm name
       - Cert ID
       - "ABA Model Rule 5.3" reference
       - Disclaimer block
     Note: plain layout for Phase 1 — branded template with logo is Phase 4
  4. Upload PDF bytes to Supabase Storage (private bucket "certificates"):
       Path: firms/{firm_id}/employees/{user_id}/{enrollment_id}.pdf
  5. Insert certificates row { user_id, enrollment_id, firm_id, storage_path,
     cert_number, issued_at, expires_at }
  6. Call Resend REST API to email the employee:
       Subject: "Your AI Compliance Training Certificate"
       Body: "Your certificate has been issued. Download it here: [link]"
       Link points to /api/certificates/{cert_id}/url (Task 11) —
       NOT a raw storage URL

Error handling: On any failure, upsert a cert_generation_queue row with
status: 'failed' and the error message — the Cron drain (Task 12) retries.

Requirements covered: CERT-01, CERT-02, CERT-03, CERT-04, CERT-05, CERT-06

TASK 11 — Certificate Download URL Route
File: app/api/certificates/[id]/url/route.ts

What: Auth-checked endpoint that returns a 60-second signed Supabase Storage
URL. Both the cert owner and any firm admin for that firm can call this.

Logic:
  1. Verify caller is authenticated (supabase.auth.getClaims())
  2. Fetch the certificates row by id
  3. Verify caller is the cert owner OR a firm admin for that firm
  4. Call supabase.storage.from('certificates')
       .createSignedUrl(storage_path, 60)
  5. Return { url }

Requirements covered: CERT-07, CERT-08, CERT-09

================================================================================

TASK 12 — Cert Generation Queue Drain (CF Cron)
File: workers/cert-worker/src/index.ts (add a cron handler)

What: CF Workers Cron Trigger running every 5 minutes. Retries any certs
that failed during Task 10.

Logic:
  1. Query cert_generation_queue WHERE status IN ('pending', 'failed')
     AND next_retry_at <= now() LIMIT 10
  2. For each row: re-run the cert generation logic (same as Task 10)
  3. On success: update status = 'completed'
  4. On failure: increment attempt_count, set next_retry_at with
     exponential backoff, cap at 5 attempts then status = 'dead'

wrangler.toml addition:
  [triggers]
  crons = ["*/5 * * * *"]

Requirements covered: AUTO-01, AUTO-02
